<h1 align="center" style="margin: 30px 0 30px; font-weight: bold;">框架介绍及开发规范</h1>

## 前言
  尽量按照框架的规范进行开发，方便团队代码维护和协调合作，提升开发效率。

## 1.框架结构
```text
com.gz     
├── common                                // 公共模块，存放与spring框架无关的代码，即无依赖性
│       └── annotation                    // 自定义注解
│       └── constant                      // 通用常量
│       └── enums                         // 通用枚举
│       └── exception                     // 通用异常
│       └── model                         // 通用实体
│       └── util                          // 通用工具类
├── framework                             // 框架核心模块，存放与spring框架相关，与系统核心功能相关的代码
│       └── advice                        // 响应数据处理
│       └── aspectj                       // 注解实现
│       └── config                        // 系统配置
│       └── datasource                    // 数据权限
│       └── domain                        // 实体类，包括与框架相关的常量类、枚举类、注解类等等
│       └── exception                     // 框架相关的异常处理
│       └── interceptor                   // 拦截器
│       └── manager                       // 管理器，包括：异步处理等等
│       └── security                      // 权限控制
│       └── util                          // 框架相关工具类
│       └── web                           // 系统核心web功能代码
├── server                                // 服务入口模块，包含：启动类和配置文件
├── biz                                   // 业务模块
│       └── demo                          // 具体业务分类小模块（按业务需求进行分类，每个分类下都要创建对应的controller、service、mapper、domain包）
│           └── controller                // 接口控制层
│           └── domain                    // 实体类管理
│                   └── convert           // mapstruct转换类，xxxConvert
│                   └── entity            // 数据表实体类，xxxDO
│                   └── qo                // 查询条件实体类，xxxQO
│                   └── req               // 请求参数实体类，xxxReq
│                   └── vo                // 响应结果实体类，xxxVO
│           └── mapper                    // 数据持久化层
│           └── service                   // 业务处理层
├── generator                             // 代码生成模块
bin                                       // 启动命令脚本
doc                                       // 框架文档
sql                                       // 数据库初始化脚本
```

## 2.业务模块开发约束
### 2.1 框架使用及项目初始化
```text
1）根据gz-server/src/main/resources下的配置文件，修改redis配置，修改mysql配置；
2）按照mysql配置创建数据库，并执行sql/GY-EP_local_20250418.sql文件；
3）启动项目即可；
4）参照 gz-biz/src/main/java/com/gz/biz/demo业务模块，创建其他业务模块。
```
### 2.2 接口开发约定
```text
1）约定所有controller接口，统一使用json格式调用，请求参数必须创建xxxReq实体，带上@RequestBody注解；
2）响应结果按xxxVO实体返回，不允许使用基本类型，允许List<xxxVO>、Page<xxxVO>，基础类型提供了对应对象VO：BooleanVO、StringVO等等；
3）接口请求参数校验，使用@Validated注解；
```
### 2.3 接口文档约定
```text
1）接口文档使用apifox，idea需要统一安装 Apifox Helper 插件，具体使用方式请查看官方文档：https://docs.apifox.com ；
2）为了保证apifox接口文档生成格式统一规范，需要按要求加上类注释、方法注释、字段注释，按照java注释规范: /** xxx */；
3）controller的类注释需要加上模块名称，格式：xxx模块/xxx接口，一般一个模块对应一个业务分类package，
    例如：com.gz.biz.demo.controller.ReturnDemoController的注释"demo模块/返回值示例接口"。
```
### 2.4 实体类分层约定
```text
1）xxxDO，对应数据库表，在domain/entity包下，命名方式：表名+DO，DO实体必须与表结构一致，不允许增减字段；
2）xxxReq，对应请求参数，在domain/req包下，命名方式：模块名+接口名+Req（可以适当缩写命名），可以使用公共的req实体，
    例如：IdReq、PageReq等等，其他情况建议根据每个接口情况，单独创建req实体，规范接口参数，易于前后端对接和维护；
3）xxxQO，对应查询条件，在domain/qo包下，普通的查询条件也可以直接使用req传参到xxxMapper.xml中，如果查询条件过多且于req
    中的参数不同时，建议创建QO实体进行查询条件封装；
3）xxxVO，对应响应结果，在domain/vo包下，命名方式：模块名+接口名+VO（可以适当缩写命名），VO实体约定仅返回接口需要的字段；
4）xxxDTO，对应业务处理层返回结果，在domain/dto包下，命名方式：自定义+DTO，DTO实体仅在业务复杂时使用，
    DTO实体主要用于业务处理层的返回结果封装，再转成VO进行接口返回，
```
### 2.5 接口统一返回
```text
1）统一返回结果，使用com.gz.framework.advice.ResultHandlerAdvice进行统一返回处理，统一返回结果为AjaxResult；
2）接口中只允许按照约定返回VO实体，不符合要求的接口调用时会抛出异常提示；
```
### 2.6 统一异常处理
```text
1）统一异常处理，使用com.gz.framework.exception.GlobalExceptionHandler 进行统一异常处理；
2）新增的异常代码，请在com.gz.common.constant.HttpStatusConstant 中新增异常代码；
3）统一业务异常使用com.gz.common.exception.BizException往上抛出，异常代码请按照HttpStatusConstant中业务异常分类进行规划；
```
### 2.7 代码生成
```text
1）在系统功能界面中，系统工具-代码生成，按照新建的数据表进行代码生成；
2）代码生成源码对应工程模块gz-generator。
```
### 2.8 代码规范
```text
1）代码规范参考《阿里巴巴Java开发手册终极版v1.3.0.pdf》；
2）约定每个接口都有独立的入参req和返回结果VO，共用req和VO实体除外；
3）不允许多个独立模块使用同一个接口，不易维护；
4）不允许多个方法共用同一个sql通过条件判断来写，不易维护；
```
